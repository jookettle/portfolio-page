마쵸봇(각주)은 2021년, 벌써 운영한 지 만으로 4년이 넘은 챗봇이에요. 처음에는 다른 지인께서 개발중인 1인 개발 프로젝트였다가 점점 프로젝트의 규모가 커지면서 제가 합류하게 되어 지금의 팀 리더자리까지 오게 되었어요. 그 사이 입소문을 타며 유저의 수가 점점 늘어나 등록된 유저의 수가 2만 명이 넘는 대형 챗봇이 되었어요.

하지만 게임을 운영하며 몇 가지 문제에 도달하게 되었는데, 초반에 개발하던 코드가 매우 길고 복잡하며 배포 로직 없이 pm2(각주)를 사용하여 가동이 되었기 때문에 업데이트와 점검, 로그 관리 시 번거로워 Docker(각주)를 도입하게 되었어요.

pm2는 분명 편하고 좋은 도구지만, 봇 덩치가 커지니까 한계가 명확하게 보였습니다. 로컬에서 개발하고 서버에 올리면 노드 버전이나 OS 환경이 미세하게 달라서 예상치 못한 에러가 터지는 일이 잦았거든요. 게다가 업데이트를 할 때마다 서버에 접속해서 `git pull`(각주)을 받고, 패키지를 다시 설치하고 프로세스를 재시작하는 과정을 매번 손으로 직접 하는 것도 꽤나 고역이었습니다.

![마쵸봇](/posts/2026-03-12-1.png)

코드가 방대해지고 복잡해지다 보니, 이런 인프라적인 부분도 단순히 손으로 세팅할 게 아니라 텍스트로 명확하게 기록하고 통제해야겠다는 생각이 들었습니다. 복잡한 원리나 환경 세팅을 글씨로 하나하나 명확히 정의해서 풀어내는 것이 오히려 관리하기도 편하고, 마쵸봇을 즐겨주시는 약간은 '너드'스러운 유저분들의 감성과도 결이 잘 맞는다고 생각했거든요.

그래서 서버 환경 자체를 코드로 짜서 구워버리는 Docker를 도입하기로 했습니다. 단순히 봇을 켜는 용도가 아니라, 2만 명의 유저 트래픽과 이미지 합성 같은 무거운 작업을 안정적으로 견뎌야 하니 Dockerfile도 꽤 꼼꼼하게 작성해야 했죠. 아래는 실제 마쵸봇을 든든하게 받쳐주고 있는 현재의 Dockerfile입니다.

```dockerfile
FROM node:22-alpine AS base

ENV PNPM_HOME="/pnpm"
ENV PATH="$PNPM_HOME:$PATH"
RUN corepack enable

RUN apk add --no-cache tzdata && ln -snf /usr/share/zoneinfo/Asia/Seoul /etc/localtime

RUN apk add --no-cache \
    cairo-dev \
    g++ \
    giflib-dev \
    jpeg-dev \
    make \
    pango-dev \
    python3

COPY . /usr/src/app
WORKDIR /usr/src/app

FROM base AS prod-deps
RUN --mount=type=cache,id=pnpm,target=/pnpm/store pnpm install --prod --no-frozen-lockfile

FROM base AS build
RUN --mount=type=cache,id=pnpm,target=/pnpm/store pnpm install --no-frozen-lockfile
RUN pnpm run build

FROM base
COPY --from=prod-deps /usr/src/app/node_modules /usr/src/app/node_modules
COPY --from=build /usr/src/app/build /usr/src/app/build
COPY .env* /usr/src/app/

RUN addgroup -g 1001 -S nodejs
RUN adduser -S nodejs -u 1001
RUN chown -R nodejs:nodejs /usr/src/app
USER nodejs

STOPSIGNAL SIGTERM

CMD [ "node", "build/run.js" ]

```

코드가 좀 길죠? 그래도 이렇게 복잡한 인프라 구조를 투박한 텍스트 쪼가리들로 완벽하게 통제할 수 있다는 게 꽤 매력적입니다.

자세히 뜯어보면 몇 가지 디테일한 포인트들이 숨어있어요. 우선 베이스 이미지는 용량 최적화를 위해 `node:22-alpine`(각주)을 썼습니다. 그리고 패키지 매니저도 무거운 npm 대신 빠르고 가벼운 `pnpm`(각주)을 쓰기 위해 `corepack`(각주)을 활성화했어요. 봇 특성상 시간에 맞춰 이벤트가 작동해야 하니 `tzdata`(각주)를 이용해 아예 컨테이너의 시간을 서울(KST)로 못 박아뒀고요.

중간에 길게 나열된 `cairo-dev`, `pango-dev` 같은 것들은 마쵸봇의 이미지 생성에 사용하는 `node-canvas`(각주)를 위한 C++ 빌드 도구들입니다. 예전에는 이걸 서버에 일일이 깔다가 에러가 나면 구글링하느라 밤을 새웠는데, 이제는 도커 파일 안에 텍스트로 적어두기만 하면 알아서 설치되니 정말 편해졌습니다.

이 도커파일의 진가는 멀티 스테이지 빌드(Multi-stage Build)(각주)에서 나옵니다. TypeScript(각주)를 컴파일하거나 패키지를 설치하는 무거운 작업은 `build` 단계에서 다 끝내버리고, 최종 이미지에는 실행에 딱 필요한 결과물(`build/` 폴더)과 운영용 패키지(`prod-deps`)만 쏙 빼와서 합칩니다. 쓰레기는 버리고 알맹이만 챙겨서 컨테이너를 가볍게 유지하는 거죠.

마지막으로 보안을 위해 root 권한 대신 `nodejs`라는 별도 유저를 만들어 권한을 낮췄고, `STOPSIGNAL SIGTERM`(각주)을 넣어서 서버가 업데이트로 꺼질 때 봇이 유저의 요청을 처리하다 말고 픽 죽어버리지 않도록, 하던 일을 마치고 종료되도록 처리했습니다.

이제는 터미널에 명령어 한 줄만 치면 이 복잡한 세팅이 완벽하게 적용된 마쵸봇이 바로 가동됩니다. "내 컴퓨터에선 되는데 서버에선 왜 안 되지?"라며 머리를 쥐어뜯지 않아도 되니 심리적인 안정감이 엄청나네요.

단순한 1인 토이 프로젝트가 어느새 2만 명의 유저를 품고, 부족하지만 안정적인 도커 환경 위에서 돌아가는 걸 보니 감회가 새롭습니다.

추후 기존 레거시로 사용중인 MongoDB(각주)를 postgresql(각주)로 이전하는 프로젝트도 진행할 예정이니 많은 기대 부탁드립니다.
